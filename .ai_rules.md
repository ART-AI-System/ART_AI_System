## 1. Branching & Git Workflow (ràng buộc đầu tiên & Nguyên tắc)
- Luôn tạo nhánh mới khi có yêu cầu "task mới:"
- Tên các nhánh luôn phải tuân theo định dạng: `feature/vietkhoa/...(tên tính năng)...`
- Tuyệt đối không được thực hiện git push, hoặc phát hành lên môi trường dùng chung (shared environment) nếu chưa có yêu cầu rõ ràng từ người dùng (từ khoá: commit code).
- Mọi thay đổi chỉ được thực hiện trong môi trường local hoặc workspace hiện tại.
- Sau khi hoàn thành code:
  - Viết code / sửa lỗi
  - Chạy test local
  - Báo cáo kết quả và Hỏi ý kiến người dùng.
  - Khi người dùng cho phép: Tự động commit code (viết commit message chuẩn), push lên remote và tự động tạo Pull Request (PR).
  - Chờ lệnh triển khai hoặc Merge.

## 2. Tiêu chuẩn Thay đổi (Change Standards)
Mọi thay đổi phải đảm bảo:
- Không làm hỏng tính năng hiện có.
- Không phá vỡ API hiện hành.
- Không làm sai dữ liệu cũ.
- Không làm giảm hiệu năng đáng kể.
- Không làm thay đổi hành vi người dùng ngoài phạm vi yêu cầu.

## 3. Đánh giá trước khi hoàn thành Task
Trước khi hoàn thành task, AI phải đánh giá:
- Frontend
- Backend
- Database
- Authentication
- Authorization
- AI Logic
- Integrations
- Analytics
- Logging
đều còn hoạt động bình thường.

## 4. PLANNING MODE
Khi người dùng sử dụng các từ khóa:
- "trao đổi"
- "phân tích"
- "thiết kế"
- "lập kế hoạch"
- "proposal"

AI phải chuyển sang Planning Mode.
Trong Planning Mode:
- Không viết code ngay.
- Phân tích yêu cầu.
- Đề xuất phương án.
- Đánh giá rủi ro.
- Đánh giá tác động hệ thống.
- Xây dựng implementation plan.
Chỉ được bắt đầu code khi người dùng phê duyệt.

## 5. HOTFIX MANAGEMENT
Khi đang có kế hoạch lớn đang thực hiện nhưng xuất hiện lỗi khẩn cấp:
Nguyên tắc
- Không được ghi đè kế hoạch hiện tại.
- Tách riêng phạm vi Hotfix.
- Chỉ sửa đúng lỗi được yêu cầu.
- Không mở rộng phạm vi công việc.
Sau khi Hotfix hoàn tất:
- Xác nhận hệ thống ổn định.
- Quay lại kế hoạch chính.

## 6. DEPENDENCY MANAGEMENT
Khi:
- Pull code mới.
- Merge branch.
- Thêm package mới.
- Nâng cấp framework.
AI phải:
- Cài đặt dependency cần thiết.
- Đồng bộ lock file.
- Kiểm tra build.
- Kiểm tra test.
Ví dụ: npm install, pnpm install, yarn install...

## 7. RESEARCH & REUSE FIRST
Trước khi viết logic mới, AI phải kiểm tra:
- Existing utilities
- Shared libraries
- Internal services
- Existing APIs
- Existing components
Ưu tiên:
- Reuse code hiện có.
- Sử dụng thư viện đã được kiểm chứng.
- Chỉ tự xây mới khi không có giải pháp phù hợp.
Không được "phát minh lại bánh xe".

## 8. TEST-DRIVEN & SECURITY FIRST
Kiểm thử
Đối với tính năng quan trọng:
- Unit Test
- Integration Test
- Edge Cases
- Regression Cases
phải được xem xét trước khi hoàn thành.

Bảo mật
Bắt buộc:
- Không hardcode secrets.
- Không hardcode API keys.
- Validate toàn bộ input.
- Chống SQL Injection.
- Chống XSS.
- Chống CSRF (nếu có web form).
- Kiểm tra quyền truy cập.
- Tuân thủ nguyên tắc least privilege.

## 9. MODULAR DESIGN
Không giải quyết các bài toán lớn trong một file khổng lồ.
Ưu tiên:
- Single Responsibility Principle
- SOLID
- Clean Architecture
- Separation of Concerns
Code phải:
- Dễ đọc
- Dễ test
- Dễ bảo trì
- Dễ mở rộng

## 10. AI AGENT OPERATING PRINCIPLE
AI phải luôn:
- Hiểu phạm vi trước khi sửa.
- Phân tích tác động hệ thống.
- Ưu tiên an toàn hơn tốc độ.
- Ưu tiên khả năng bảo trì hơn giải pháp tạm thời.
- Giữ tương thích ngược khi có thể.
- Minh bạch về rủi ro và giả định.

Mặc định:
"Nếu không chắc chắn về mức độ an toàn của thay đổi, AI phải dừng lại và xin xác nhận từ người dùng trước khi tiếp tục."

## 11. Ứng dụng Graphify (Knowledge Graph)
* Công cụ graphify được dùng để biểu đồ hoá các file thành dạng knowledge graph.
* Thay vì để các trợ lý AI phải đọc lại toàn bộ tài liệu từ đầu mỗi khi đặt câu hỏi, Graphify sẽ phân tích dự án một lần duy nhất và lưu các kết nối dưới dạng đồ thị ngay trên máy.
* Đặc điểm: Tối ưu hóa chi phí AI, Hỗ trợ đa phương thức (code, PDF, hình ảnh, sơ đồ), Bảo mật dữ liệu (cục bộ).
* Ứng dụng vào code: Tránh trường hợp lỗi chồng chéo logic.

## 12. Bài học từ ECC (Core Skills bổ sung)
* **Research & Reuse First:**
  * Kiểm tra existing utilities, shared libraries, internal services trước khi tự code mới. Không "phát minh lại bánh xe".
* **Database Strategy (Native MongoDB):**
  * Bắt buộc sử dụng Native MongoDB Driver (`mongodb` package) thay vì Mongoose. Xây dựng các schema class thuần tuý tương tự code mẫu có sẵn.
* **Immutable Operations:** (Học từ ECC) Luôn ưu tiên cập nhật bất biến thay vì thay đổi trực tiếp state dùng chung. Không bao giờ bypass các validation hooks hoặc security checks đã được thiết lập.
* **Delegation & Role Segregation:** Không xử lý tất cả trong một luồng duy nhất nếu không cần thiết. Đối chiếu logic với Code Architect, Security Reviewer và Test Engineer.
